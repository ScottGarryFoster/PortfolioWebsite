Looking at the grid

Personal Project

The goal of the project was to produce working A star pathfinding completely with my own code.

The foundations of this was to produce a grid to traverse and in order to do this I decided upon a simple three step method. Start with a grid of collisions marked with # symbols and draw a vague line from the top left to the bottom right, then top right to bottom left and finally just draw a few random lines in the middle. The result is as shown in the image, a random grid which occasionally has dead ends and often has alternative routes to the target.

The code shown is the vague drawing between two points. If I draw the line too well it’s not really a challenge, if it’s all random then the variance is huge. To fix this I have the loop which calls this send in its iterator and every other time it’ll head directly towards the target.

To make sense of the difference between targetDirection and randomDirectionNumber: targetDirection is the vague direction and can be found on the number pad, so of the target is to the top left targetDirection will be 9, randomDirectionNumber is a random direction and will only allowed if it’s not be moved in before or if it agrees with targetDirection.

The main loop

The pathfinding itself begins with finding a path between two points.

Currentpoint is used to store the point we’re interested in and we begin with the origin. The loop occurs until I find the target or there’s nothing on the openlist left to search.

The first thing within the loop is check all the directions around the point we are at, this method will check to see if there is a collision in which case that tile can go on the closed list, if it’s on the closed list it won’t go on the open list again and if it’s not a collision it’ll go on the open list as not explored. When things go onto the open list the costs are calculated.

I removed the current point from the open list and add it to the closed list, so we don’t look into it again.

Then I search the open list to find the lowest F cost of an explored tile. If two have the same F cost, then I find the lowest H which is just the distance from the target if there is no other collisions in the way.

If I find nothing there is no path and we return nothing. Also, before we check the open list if this is blank for whatever reason I don’t need to check the open list it’ll just return a blank list. Finally, we break out of the loop if we have the target and follow the path back by following the parent and return the completed list.

At the bottom I’ve pasted in the Pathfinding Costs just for context of the class.

Unexplained code and interface

Something yet to be explain is hidden behind Validate new direction which simply calls a method to detect if the test is in the grid or a collision, if so, add it uniquely to the closed list. If not and it’s not on the close list add it uniquely to the open list.

When it comes to calculating H cost, if I were to make diagonal movement possible, I’d use the distance between two vector calculation: A(squared) + B(squared) = C(Squared). Due to the idea that all I need is the cost it’d take to move straight to the target if there were no collisions though I can save a lot of memory space by simply finding difference between the vectors.

Within the file found here is the main method where the screenshot code is run. This brings everything together. If you enter the number 1, a new maze is generated. If you enter 2, you can move the player or @ symbol, and if a path can be found a path will be attempted. If you enter a number outside of the grid it’ll just keep you where you are.

That is basically this pathfinding code, all of this is free to download and compile and was created with Visual Studio 2019 and maybe found here.

Reflection on this project

Before university I’d tried A Star pathfinding in RPG Maker and failed many times and barely had a working version which used a lot of memory and was not efficient.

When coming to university something I was keen on was attempting it with my Pacman project and looking back on the code I wrote at the start of my course now at in my second year I’m not as embarrassed as I think I would be as I feel that pathfinding is just a skill you need to get better at.

Having made pathfinding in C# with lists I attempted to make this work with C++ lists however honestly I feel like the code isn’t that much smaller than it was in my Pacman. I’m happy with the main method but it took a lot of setup to get to the end point, there are far more classes than perhaps are needed, far more enums and far more memory.

I’m not making this project to become better at Mazes, maybe that’d be a project for the future, but the grid generation system for what it is I like. It’s simple and gets the concept out there the only issue I have is the constant confusion over row and y along with col and x. I need to make these concepts more consistent and plan ahead for these being consistent.

Overall happy even if I know with more planning I could do it far better.
